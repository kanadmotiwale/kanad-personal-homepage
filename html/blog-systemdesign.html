How I Think About Scalability

March 2026 · 6 min read

Scalability is one of those words that gets thrown around a lot in software engineering interviews and architecture discussions, but the actual thinking behind it is more nuanced than most people let on. Here is how I approach it after building systems that had to grow.

Scalability is Not Just About More Users

The most common mental model for scalability is handling more traffic, but that is only one dimension. When we were scaling FastFindFirm from a few hundred users to several thousand, the bottlenecks were rarely the application servers themselves. They were the database query patterns we had not thought through, the background jobs competing for resources, and the third-party APIs we were hammering without rate limiting.

Good scalability thinking means anticipating where the system will hurt before it actually does. That requires understanding the full data flow, not just the happy path.

Vertical vs Horizontal Scaling

Vertical scaling means giving your existing server more resources: more CPU, more RAM, more storage. It is fast to implement and requires no code changes, which makes it appealing in the short term. The problem is that it has a ceiling and it creates a single point of failure.

Horizontal scaling means adding more instances and distributing the load between them. It is more complex to implement correctly because you have to think about statelessness, session management, and distributed consistency, but it is the only approach that scales without limits in theory.

The Database is Almost Always the Bottleneck

After working on several backend systems, I have learned that the database is almost always where performance problems show up first. Poorly indexed tables, N+1 query patterns, and transactions held open for too long will hurt you long before your application code becomes a problem.

The things that have helped me most are writing queries that use indexes properly, using connection pooling aggressively, adding read replicas for read-heavy workloads, and being very deliberate about what actually needs to be in a relational database versus a cache or a document store.

Design for Failure from the Start

Systems that scale well are not just fast, they degrade gracefully. Circuit breakers, retry logic with exponential backoff, timeouts on every external call, and fallback behaviors for non-critical features are the difference between a system that handles a bad day and one that goes down completely.

The most valuable lesson I took from working on enterprise software is that you cannot predict every failure mode, but you can build systems that contain failures instead of propagating them. That mindset, more than any specific technology choice, is what makes a system truly scalable.
